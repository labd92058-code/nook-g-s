## TASK 4 — BILLING ENGINE AUDIT & DUAL BILLING MODE

### 4A — Audit existing time billing logic

Review the entire time billing engine and verify:
- Session start time is recorded correctly (use server timestamp, not client time)
- Session duration is calculated without floating point drift
- Billing rate is applied consistently (per minute or per hour — pick one and enforce it everywhere)
- Paused sessions: if pause is supported, verify pause/resume time tracking is gap-free
- Session end: verify the final amount due is calculated correctly and matches what gets written to balance_transactions
- Edge cases handled: session ending at midnight, very short sessions (under 1 minute), very long sessions (over 8 hours)
- No session can end with a negative duration
- The billed amount must always be a non-negative number rounded to 2 decimal places

Flag any calculation bugs found with a // BILLING BUG comment and fix them.

### 4B — Build dual billing mode

When a new session is opened for a client, the staff member must choose one of two billing modes. This choice is made once at session start and cannot be changed mid-session.

---

MODE 1: TIME BILLING
- The client pays based on how long they sit
- Rate is set per venue (e.g. 5 MAD per 30 minutes)
- Timer starts when the session is opened
- At checkout, the total due = time elapsed × rate
- Any consumptions ordered during the session are tracked for internal records only — they do NOT add to the final bill
- Revenue recorded = time amount only

---

MODE 2: CONSUMPTION BILLING
- The client pays only for what they order (coffee, food, etc.)
- Time is still tracked and displayed on the session (for operational awareness) but time is NEVER billed
- At checkout, the total due = sum of all products ordered
- Time cost must NEVER appear in the invoice, receipt, or balance_transactions entry
- Revenue recorded = consumption amount only

---

Implementation requirements:

Database:
- Add a billing_mode column to the sessions table: ENUM or CHECK constraint, values: 'time' | 'consumption'
- Add a session_consumptions table if it does not exist: id, session_id, product_id, quantity, unit_price, created_at
- The balance_transactions entry at checkout must record: amount, billing_mode, and a breakdown field (JSONB) with the detail

Session open flow:
- When opening a session, show a clear modal/step where staff selects billing mode
- Display a short explanation of each mode so staff cannot confuse them
- Store the chosen mode in the session record immediately on creation

Checkout flow:
- If billing_mode = 'time': calculate time cost, show time breakdown, set consumptions aside in the breakdown JSONB but do not add to total
- If billing_mode = 'consumption': sum all session_consumptions, show itemized receipt, never show or calculate a time charge
- The checkout UI must visually distinguish the two modes (different badge color or label)
- After checkout, session status = 'closed', end_time recorded, final_amount stored

Types to create:
  type BillingMode = 'time' | 'consumption'

  type Session = {
    id: string
    client_id: string
    venue_id: string
    billing_mode: BillingMode
    start_time: string        // ISO timestamp, server-generated
    end_time: string | null
    status: 'open' | 'closed' | 'paused'
    final_amount: number | null
    created_at: string
  }

  type SessionConsumption = {
    id: string
    session_id: string
    product_id: string
    quantity: number
    unit_price: number
    created_at: string
  }

  type BalanceTransaction = {
    id: string
    client_id: string
    venue_id: string
    amount: number
    billing_mode: BillingMode
    breakdown: Record   // JSONB
    created_at: string
  }

## TASK 5 — OFFLINE MODE

Build a complete offline-first layer so Nook OS works fully when the internet connection drops (e.g. unstable café wifi).

### Strategy
- Use IndexedDB (via idb or Dexie.js) as the local cache and offline queue
- Detect connectivity using navigator.onLine + a lightweight Supabase ping
- Show a persistent offline banner in the UI when the app is offline

### What must work offline
- Opening a new session (saved locally, synced when online)
- Adding consumptions to an open session (saved locally)
- Closing/checking out a session (calculated locally, synced when online)
- Viewing currently open sessions (read from local cache)
- Viewing the product list (read from local cache, refreshed when online)

### What can be deferred until online
- Creating new client accounts
- Viewing historical reports or past transactions
- Changing venue settings

### Sync logic
- Every write made offline is added to an outbox queue in IndexedDB with: action type, payload, timestamp, retry count
- When connectivity is restored, process the outbox queue in order (FIFO)
- If a sync fails after 3 retries, mark the entry as failed and alert the admin
- After successful sync, clear the local outbox entry and confirm with a toast notification

### Conflict resolution
- If a session was opened offline and the same client already has an open session online, flag it as a conflict and require manual resolution by admin
- Never silently overwrite server data — always prefer server data for reads, local data for pending writes

### Implementation
- Create /lib/offline/db.ts — initializes the IndexedDB schema using Dexie
- Create /lib/offline/queue.ts — outbox queue: enqueue, dequeue, retry logic
- Create /lib/offline/sync.ts — sync runner triggered on connectivity restore
- Create /shared/hooks/useConnectivity.ts — returns { isOnline, lastSyncAt }
- Wrap all Supabase write calls in a helper that decides: if online → write to Supabase directly; if offline → write to local DB + add to outbox
